01

The problem

Vendia Share's pricing model charges clients based on platform activity — reads, writes, shares, subscriptions, and storage across their network of nodes. But clients had no way to see this data themselves.

Every billing query required a support ticket, fielded manually by solution architects. As Vendia transitioned to a credit-based model, this process became unsustainable — clients couldn't predict spend, couldn't optimize usage, and couldn't make informed budget decisions.

Before

How it worked

Monthly PDF invoices emailed post-billing.

No real-time consumption visibility.

Every query routed through support tickets.

Manual work for solution architects each month.

After

What we shipped

Self-serve dashboard, live consumption data.

Credit and cost visibility at Uni / node level.

Zero support ticket overhead for billing queries.

Budget forecasting enabled for the first time.

02

Research & discovery

I ran 5 stakeholder interviews, 3 user interviews, and a cross-functional design workshop — targeting two distinct audiences with very different goals.

Business leaders needed portfolio-level cost summaries to plan budgets and justify spend to their organizations. Engineers needed granular, node-level breakdowns to detect anomalies and understand usage patterns in real time.

This tension — summary vs. granularity — became the central design constraint I had to resolve without overwhelming either group.

Design workshop artifact — cross-functional alignment session

Key finding

Credit ≠ cost (yet)

Users understood “activities” but not how they mapped to credits or dollars. The translation layer needed to be explicit in the UI, not assumed.

Key finding

Prediction is the goal

Users didn't just want to see past spend — they wanted to forecast future usage to stay within budget. Historical data needed a forward-looking frame.

Key finding

Portfolios get complex fast

Large clients manage dozens of Unis and Nodes. Any design that surfaced entity-level data had to scale gracefully or it would collapse under real-world complexity.

How might we

“How might we create a consumption dashboard that supports the changing pricing structure, while being flexible to accommodate users with different engagement models?”

03

Three pivots that shaped the design

Rather than one linear path to a solution, this project was defined by three inflection points — moments where early assumptions broke and the design had to evolve.

Pivot 01

From one dashboard to two

The first iteration combined activity data and billing data on a single screen. It looked complete on paper, but usability testing exposed a fundamental conflict: usage metrics (counts of reads/writes) and financial metrics (credits, currency) operate on completely different cognitive registers. Users couldn't orient themselves.

I made the call to split the experience into two purpose-built dashboards: Consumption (what did I do?) and Billing (what did it cost?). This added scope — but resolved the core confusion and made both views significantly more useful.

Iteration 1 — mixed metrics causing orientation issues

Pivot 02

Evaluating six visualization tools — then building within constraints

Iteration 2 — toolbar component enabling data slicing

Before designing a single chart, I evaluated six data visualization tools (Tableau, DOMO, Mode, and others) against a rigorous set of requirements: white-labeling capability, Vendia's RBAC logic, row/field-level access control, query performance, and accessibility.

I prototyped initial visualizations in DOMO using real client data — which gave stakeholders something concrete to react to early. When the engineering team ultimately built the final implementation natively, my DOMO prototypes served as the functional specification. The tool evaluation process also surfaced a critical constraint: the visualization layer needed to mirror Vendia's permissions model exactly, which shaped how I structured every filter interaction.

Pivot 03

Solving the “too many entities” problem with a toolbar system

Providing granular data by Uni and Node was a firm stakeholder requirement. But Iteration 2 revealed a usability ceiling: clients managing 20+ Unis found the entity-level views overwhelming and unusable.

The solution was a new Toolbar component I designed for the existing design system — a persistent, context-aware filtering layer that let users slice to any entity without losing their place in the dashboard. I documented 5 component states with detailed specifications, enabling engineering to implement it directly into the product library. This became the architectural foundation for both dashboards.

05

Solution highlights

Table component — bidirectional scrolling, spreadsheet-familiar patterns

Familiarity as a design principle

The core table component was designed to mirror spreadsheet conventions — bidirectional scrolling, sortable columns, and inline comparisons. This was a deliberate choice: the primary users were finance and engineering teams fluent in Excel. Reducing the learning curve was as important as the data itself.

Share Summary — the uniquely Vendia view

One visualization had no equivalent in competing tools: the Share Summary, surfacing who shared data with whom, and which nodes participated in each exchange. This directly reflected Vendia's core value proposition and gave users a uniquely meaningful view of their network activity — not just raw counts.

07

Reflections

Embed with engineering early

Weekly syncs with developers from week one meant technical constraints surfaced during ideation — not during handoff. When the DOMO prototype gave way to native implementation, there were no surprises. This kind of collaboration isn't a nice-to-have; it's how complex data products actually get built well.

Real data reveals what mockups hide

Prototyping in DOMO with an actual client's usage data exposed edge cases and scale problems that lorem ipsum never would have. Testing against the messiness of real-world data earlier — even imperfectly — saved significant rework downstream.

Splitting scope can be the right call

Separating one dashboard into two felt like scope expansion. But it was the right design decision — and selling it required stakeholder alignment built on research evidence, not instinct alone. Advocating for complexity when the user need justifies it is a senior designer skill.

Design system contributions compound

The 3 components I contributed to the design system weren't just project deliverables — they became reusable infrastructure. Designing with the library's future in mind, not just the immediate screen, is how individual design work multiplies across the product.

08

What I'd do next

An AI-powered suggestion layer

The dashboard surfaces what happened. The natural evolution is a layer that tells you what to do about it.

An AI-powered suggestion model embedded in the dashboard could analyze individual usage patterns and surface tailored recommendations — flag an underutilized node, suggest consolidating two Uni networks, or alert before a client's credits run out. This shifts the product from a reporting tool to an optimization tool.

For Vendia, this creates a compounding business opportunity: clients who actively optimize their usage get more value from the platform, and the actionable insights drive deeper product engagement. The dashboard becomes a retention and expansion lever — not just a transparency feature.

Opportunity areas
  • Predictive credit alerts: Warn clients before they overshoot their monthly budget — not after the invoice arrives.
  • Personalized optimization tips: Tailor recommendations to each client's engagement model — not a generic best-practice checklist.
  • Actionable product insights: Identify underused features and growth opportunities — turning the dashboard into a PM tool as much as a user tool.
